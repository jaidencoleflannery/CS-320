# Ensuring Functional, Secure Software

Probably the most important thing to software development is developing reliable and understandable tests that balance the line of exhaustion and loose insurance. Unit tests with JUnit verify that the user’s convention of units (usually individual methods) and their related classes behave as you would expect and don’t change in the future, which could ruin any other methods that rely on the code’s validations. Each test should isolate a “piece” of logic, assert an outcome from whatever action you perform, and if it fails, you know and can fix it. Functionality alone isn't enough sometimes, security requires endless effort - validating input, no hardcoded credentials, and handling exceptions like a sane person are all the core of writing decent, reliable software. Static analysis tools and code reviews can help with this, and they add additional layers of assurance, but in the end a great developer understands that testing should be taken piecewise, and each type of testing should cover a different specific need; unit tests should cover validations and constant values that other methods might rely on, so that you cannot silently break functionality without realizing it - regression and manual testing should be used to make sure flow works properly, and your “units” interact properly - integration testing should be used to make sure the eventual outcome, or what is pushed out and into your program, reacts and acts exactly to spec. 
Testing and secure coding generate software that actually works, and can be worked on.

# Interpreting and Incorporating User Needs

Understanding users is one of the hardest challenges (next to naming variables). Actually parsing needs versus wants is almost an impossible task, especially if you work in a heavily specialized niche field. Requirements gathering can help with this, as it starts with a baseline so that your software is minimally usable, and further functionality is non-critical and can be properly weighed. You typically want to ask requirement questions like: What problem is being solved? Who will use the software, and in what context? Can I trust my user’s opinions? (half joking)
From there, “stories” can be turned into testable behavior that we can model our code around. Each test can map directly to a requirement, and we can do a heavy restriction on all other functionality to minify our scope. It serves as documentation of what the program is supposed to do, at least for the time being. When a test passes it confirms that a specific user need has been met.

# Approaching Software Design

Good design is a conspiracy theory, but we can produce great software in the pursuit of it. It should start with breaking a problem into smaller components, and understanding what problem you actually need to solve in that subsystem. OOP principles make this incredibly easy, and almost unavoidable in small projects; classes should be loosely coupled, polymorphic and extensible. Planning the structure via diagrams, or simply outlining basic relationships prevents rework and is good for any project, even small tools. Design, really, at the end of the day largely means maintainability as well - most developers can build a script that works but is hell to work on, very few great developers can write a script that works and is easy to augment. Code that is easy to test is a good marker for well written code, it means there are clear interfaces and minimal hidden dependencies that aren’t obviously apparent. Writing unit tests with your code naturally pushes cleaner, modular software, which we always need more of.
